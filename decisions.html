Architectural Decisions — 郵件收發管理系統

Last reviewed: 2026-04-09 · Reviewer: Gemini (via ai-review skill)

ID Decision Status Rationale & Notes
D1 Component State Structure MODIFY Original: Flat 12+ useState calls inside one MailTable component.
Change (2026-04-09): Extracted modals into standalone memoized components (ConfirmActionModal, PickupModal, EditMailModal). Each modal manages its own internal state (form, loading). MailTable retains only "which modal is open" state and batch/row-selection state.
Why: 12+ states in one component caused hard-to-track side effects and unnecessary re-renders across unrelated UI sections.
D4 Batch Operation Error Handling MODIFY Original: Promise.all — fails fast; one failure aborts all remaining items.
Change (2026-04-09): Replaced with Promise.allSettled. Results are counted and reported via notification.warning ("成功 N 筆，失敗 M 筆") so operators know exactly which items need attention.
Why: With 50 items, one transient failure previously left the system in an inconsistent state with no per-item visibility.
D6 Column Filter Date Computation MODIFY Original: dayjs() called inside onFilter — executed per-row on every keystroke.
Change (2026-04-09): useMemo pre-computes _receivedFmt and _deadlineFmt strings when the items prop changes. onFilter performs a simple string includes().
Why: O(N) date parsing per keystroke causes UI lag on large datasets. Pre-computing shifts cost to data-load time (once), not filter time (continuous).
D7 Modal / Form Lifecycle MODIFY Original: Mixed lifecycle — pickupForm used destroyOnClose but editForm did not, risking stale field values between different mail items.
Change (2026-04-09): All modals use destroyOnClose. Extracted modals create their own Form.useForm() internally and use initialValues (derived from item prop) which are applied fresh on each mount.
Why: Consistent teardown eliminates "ghost data" from previously viewed items appearing in a newly-opened edit form.
D8 API Service Layer ADOPT Decision (2026-04-09): All fetch() calls extracted into src/services/mail.ts (mailApi.notify, mailApi.put, mailApi.delete, mailApi.uploadPhoto, mailApi.ocr).
UI components import mailApi — no inline fetch() in JSX files.
Why: Inlined fetch calls make components impossible to unit test without heavy global.fetch mocking. A dedicated service module can be mocked at the module boundary. Also centralises error-response parsing (unwrap() helper).
